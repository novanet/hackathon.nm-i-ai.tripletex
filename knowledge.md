## Tripletex API Quirks

- **`GET /ledger/voucher` requires `dateFrom` + `dateTo`, NOT `voucherDateFrom` + `voucherDateTo`** ŌĆö repeated 422s in external agent logs confirmed the list endpoint rejects the voucher-prefixed parameter names even though they look plausible. Relevant for ledger-correction and month-end analysis flows. _(2026-03-22)_

- **`GET /ledger/posting` still requires `dateFrom` + `dateTo` even when filtering by `voucherId`** ŌĆö calling `/ledger/posting?voucherId=...` without a date window returns 422. Correction/debug handlers should always include an explicit bounded date range. _(2026-03-22)_

- **`/ledger` endpoint does NOT return `account.number` or `account.name` by default** ŌĆö must explicitly request `fields=account(id,number,name),sumAmount`. Without this, `account.number` defaults to 0 and all expense account filtering fails silently. This was the root cause of task 28 (cost*analysis) scoring 0/5 in competition. *(2026-03-22)\_

- **Employee `startDate` is NOT a field on `/employee`** ŌĆö use `POST /employee/employment` as a separate call after creating the employee. The employee POST will 422 with "Feltet eksisterer ikke i objektet" if you include `startDate`. _(2026-03-20)_
- **Employee `dateOfBirth` must be set** before creating employment ŌĆö otherwise 422 "Feltet m├ź fylles ut". Always extract DOB from prompt. EmployeeHandler now defaults to `1990-01-01` if not provided. _(2026-03-20, UPDATED 2026-03-21)_
- **Employee `userType` must not be `"0"` or empty** ŌĆö use `"STANDARD"`. Error: "Brukertype kan ikke v├”re ┬½0┬╗ eller tom." Must be set when creating employees anywhere (specifically in TravelExpenseHandler and other handlers, not just EmployeeHandler). _(2026-03-20)_
- **Employee `department.id` required when department module active** ŌĆö query `GET /department?count=1` and use first available ID. If none exists, EmployeeHandler now auto-creates a default "Hovedavdeling" department. Sandbox and competition both have department module enabled. _(2026-03-20, UPDATED 2026-03-21)_
- **Employee `email` is required** for Tripletex users ŌĆö 422 "M├ź angis for Tripletex-brukere" if missing. Use `userType = "NO_ACCESS"` when creating employees without an email (e.g. in PayrollHandler auto-create). NO*ACCESS employees don't need email and can still receive payroll. *(2026-03-20, UPDATED)\_
- **Employment `division` required in ALL environments** ŌĆö BOTH sandbox and competition require `division.id` on employment for payroll. `POST /employee/employment` may silently ignore the `division` field, so `EnsureEmploymentDivisionLinkedAsync` now does a belt-and-suspenders PUT if the division is missing after POST, then re-verifies. Without this, `POST /salary/transaction` fails with "Arbeidsforholdet er ikke knyttet mot en virksomhet". _(2026-03-22, UPDATED)_
- **Employee PDF reruns in sandbox hit duplicate email collisions** ŌĆö when `POST /employee` returns 422 `email: Det finnes allerede en bruker med denne e-postadressen.`, reuse the existing employee if first/last name match; otherwise retry once with a unique email. This preserves validator expectations for dirty sandbox runs while staying harmless in clean competition environments. _(2026-03-21)_
- **Employment details are keyed by date per employment** ŌĆö rerunning `POST /employee/employment/details` for the same `employment + date` returns 422 `date: Finnes fra f├Ėr.`. Treat employee onboarding details as upsert/idempotent for sandbox replays. _(2026-03-21)_
- **Project `startDate` is required** ŌĆö defaults to today if not specified in prompt. Error: "Feltet m├ź fylles ut." _(2026-03-20)_
- **Project manager needs entitlements** ŌĆö must call `PUT /employee/entitlement/:grantEntitlementsByTemplate?template=ALL_PRIVILEGES` before assigning as PM. Error: "Oppgitt prosjektleder har ikke f├źtt tilgang som prosjektleder i kontoen". _(2026-03-20)_
- **Invoice requires company bank account** ŌĆö the company (not customer) must have a bank account registered. Error: "Faktura kan ikke opprettes f├Ėr selskapet har registrert et bankkontonummer." Seen in competition but not sandbox. _(2026-03-20)_
- **Voucher requires `postings` array** ŌĆö cannot POST without it. Error: "Et bilag kan ikke registreres uten posteringer." _(2026-03-20)_
- **Multi-voucher support** ŌĆö When prompts request multiple vouchers (corrections, year-end closings), the LLM extracts `voucher1`, `voucher2`, etc. instead of a single `voucher` entity. VoucherHandler now detects keys matching `voucherN` (where N is a digit), iterates each, builds postings per-voucher, and POSTs separately. Two extraction formats observed: (1) `debitAccount`/`creditAccount`/`amount` pairs, (2) structured `postings[]` array with `debitCredit` field. `BuildPostingFromJson` now handles `debitCredit`: "credit" negates the amount. Non-numeric amounts (e.g. "calculated*tax_cost") are safely skipped. Per-voucher try/catch prevents one 422 from crashing the entire batch. *(2026-03-21)\_
- **Department numbers collide** ŌĆö pre-query existing departments with `GET /department?fields=departmentNumber,name`, then pick the next available number. _(2026-03-20)_
- **Customer lookup by org number** ŌĆö use `?organizationNumber=` param, NOT `?name=` with the org number as value. _(2026-03-20)_
- **Timesheet entry requires `activity`, `date`, `employee`** ŌĆö minimum fields. Also supports `project`, `hours`, `comment`. Only one entry per employee/date/activity/project combo. 38 hours on a single date works (no 24h cap on `hours` field). _(2026-03-20)_
- **Activity creation requires `activityType`** ŌĆö use `PROJECT_GENERAL_ACTIVITY` for project-related billable activities. Link to project via `POST /project/projectActivity`. _(2026-03-20)_

- **LLM entity naming variants** ŌĆö entity keys like `payment` may be named `payment1`, `invoice` may be `invoice1`, `voucher` may be `voucher1`. Always check both bare name and numbered variant (`payment ?? payment1`). Detection logic must check all variants. _(2026-03-21)_
- **Account 1500 (Kundefordringer/Receivables) is a system control account** ŌĆö manual `/ledger/voucher` postings against it are rejected with 422 "systemgenererte og kan ikke opprettes eller endres p├ź utsiden av Tripletex." Same likely applies to 2400 (Leverand├Ėrgjeld) and other balance sheet control accounts. Make voucher creation non-fatal for reminder-fee tasks ŌĆö the reminder fee invoice creates the implicit receivables entry automatically. _(2026-03-21)_
- **Dynamic VAT type lookup for orders: use `typeOfVat=OUTGOING` and prefer number `"3"`** ŌĆö when hardcoded `DefaultOutputVatTypeId=3L` fails with 422 "Ugyldig mva-kode", do `GET /ledger/vatType?typeOfVat=OUTGOING&count=100&fields=id,number,percentage` and pick the entry with `number=="3"`. This is InvoiceHandler's `ResolveVatTypesFull` method ŌĆö reuse it via `_invoiceHandler.ResolveVatTypesFull(api)` (it's `internal`). _(2026-03-21)_
- **Current local sandbox exposes only one active outgoing VAT type: `id=6`, `number=6`, `0%`** ŌĆö a direct `GET /ledger/vatType?typeOfVat=OUTGOING&fields=id,number,name,percentage` on 2026-03-22 returned only `No output VAT (outside the VAT Act)`. That means local smoke tests cannot prove 25% invoice totals for Task 29 and similar order/invoice flows; treat sandbox amount mismatches cautiously when the handler is otherwise creating the correct chain. _(2026-03-22)_
- **`physicalAddress` on customer/supplier requires `country` to persist** ŌĆö when POSTing `physicalAddress: {addressLine1, postalCode, city}` without a `country: {id}`, Tripletex may silently drop the address. The country defaults to Norway (id=161) if you add `country: {id: 161}`. However, even without country in the POST body, Tripletex stores the address correctly (tested) ŌĆö the bug was in the validator's GET query. _(2026-03-21)_
- **Validator `fields=*` does NOT expand nested objects** ŌĆö when fetching e.g. `/customer/{id}?fields=*`, nested objects like `physicalAddress` are returned as id-only references. To get all nested fields, use explicit expansion: `fields=id,name,email,organizationNumber,physicalAddress(*),postalAddress(*)`. Fixed SandboxValidator.cs `ValidateCustomer` method. _(2026-03-21)_
- **Product `vatType` on POST is invalid in sandbox** ŌĆö local sandbox rejects ANY vatType reference on `POST /product` (Ugyldig mva-kode). Do NOT include `vatType` in product creation body ŌĆö it is optional in Tripletex and the order line's `vatType` controls invoice VAT independently. _(2026-03-21)_
- **~~Reminder fees partial payment must use `amountOutstanding`~~** ŌĆö OUTDATED (2026-03-21). Competition actually checks `correct_paid_amount = partialPaymentAmount` from the prompt, NOT `amountOutstanding = 0`. Paying prompt-specified 5000 is correct. _(corrected 2026-03-22)_
- **Overdue invoice selection must check `invoiceDueDate < today`** ŌĆö competition environments have multiple invoices; picking the first unpaid one (which may not be overdue) caused 3/6 check failures. Fixed: iterate all invoices, select the one with `invoiceDueDate` in the past (oldest overdue preferred). Fall back to first unpaid if no truly overdue invoice found. _(2026-03-22)_
- **FallbackAgent `HandlePost` body crash** ŌĆö `args.GetProperty("body")` throws `KeyNotFoundException` when LLM omits the body key. Changed to `args.TryGetProperty("body", out var b) ? (object)b : new Dictionary<string, object>()`. _(2026-03-21)_
- **Reminder fees (Task 25) routing fix** ŌĆö Root cause: `IsReminderFeeTask()` required `voucher1` entity with debit/credit accounts, but real prompts ("Finn den forfalte fakturaen og registrer purregebyr p├ź 50 NOK") produce no `voucher1`. Fix: added `reminder_fee` as explicit task*type in LlmExtractor system prompt + TaskRouter + updated `IsReminderFeeTask()` to check `extracted.TaskType == "reminder_fee"` first. Local validation: 8/8 (100%). *(2026-03-21)\_

## LLM Extraction Pitfalls

- **firstName/lastName sometimes not extracted** ŌĆö happened with German, Spanish, Portuguese, and Nynorsk prompts. The LLM returned only `email` without name fields. Root cause: extraction prompt needs explicit required-field emphasis. _(2026-03-20)_
- **Receipt PDF vouchers must keep the requested line item, not the labeled total** ŌĆö when the prompt names a specific receipt item (`Togbillett`, `USB-hub`, `Oppbevaringsboks`, etc.), keep that item description and amount and skip labeled-total normalization. PdfPig may concatenate item name and amount without spaces (`Togbillett7950.00 kr`), and German prompts also appear as `wir benotigen die X-Ausgabe aus dieser Quittung`, so prompt matching and line-item regex must handle those forms. _(2026-03-22)_
- **Employee entity sometimes missing email** ŌĆö even when clearly stated in the prompt (Portuguese prompts). Ensure extraction schema marks email as required. _(2026-03-20)_
- **Full JSON object serialized as query param** ŌĆö bug where the entire employee JSON was URL-encoded into `firstName=` and `lastName=` search params instead of just the string values. Check handler code carefully when using extracted nested objects. Use `GetScalarString` (returns null for Object/Array types) instead of `GetStringField` when extracting name/email fields that may be nested objects. _(2026-03-20)_
- **Travel expense misrouted to VoucherHandler** ŌĆö extraction returned `create_voucher` instead of `create_travel_expense`. Ensure LLM system prompt clearly distinguishes the two task types. _(2026-03-20)_
- **Handler name logging race condition** ŌĆö `TaskRouter.LastHandlerName` was shared singleton state, corrupted by concurrent requests. Fixed: `RouteAsync` now returns handler name in tuple. Previous competition logs showing "TravelExpenseHandler" for employee tasks were false ŌĆö actual routing was correct. _(2026-03-20)_
- **Composite task extraction varies** ŌĆö LLM may put employee/activity data under `timeRegistration` entity OR as separate `employee`/`project.activity` entities. Handler must check both paths. _(2026-03-20)_
- **Voucher dimension nested in voucher entity** ŌĆö LLM sometimes puts `"dimension": {"name": "Region", "value": "Vestlandet"}` inside the voucher entity instead of as a separate `"dimension"` entity. VoucherHandler has fallback to extract it from `voucher["dimension"]`. Also check singular `"value"` vs plural `"values"` key. _(2026-03-20)_
- **LLM hallucinate `vatIncluded: true`** ŌĆö when prompts don't mention VAT at all ("a 21950 NOK"), the LLM sometimes set `vatIncluded: true`, causing prices to go into `unitPriceIncludingVatCurrency`. This makes invoice total = sum of stated prices (no VAT added), but competition expects excl. VAT pricing with VAT applied on top. Fixed: LLM prompt explicitly says "do NOT set vatIncluded unless prompt explicitly says prices include VAT." Code safeguard: `BuildOrderLines` checks raw prompt for VAT-inclusive keywords before trusting `vatIncluded=true`. _(2026-03-20)_
- **FX payment extraction can collapse foreign invoices into NOK totals** ŌĆö Ironbridge replay showed `payment.currency = NOK`, `payment.amount = 136`, and missing exchange-rate fields for a prompt that clearly stated `11671 EUR` at `11.22/11.71 NOK/EUR`. `PaymentHandler` and `SandboxValidator` now recover foreign amount/currency/rates from the raw prompt + relationships when FX extraction degrades. _(2026-03-21)_

## Scoring & Validation Insights

- **Task 26 is not unknown: it is the `annual_accounts` monthly-close variant** ŌĆö raw competition logs show tx*task_id 26 improved 0 -> 2.10 -> 2.55 and the scoring run used `AnnualAccountsHandler` with a Spanish month-end close prompt (`run_id=2b2ffac8`, submission `e5c6b5db-e1f0-40cd-87c9-f7d8ce238eeb`). Keep Task 26 separate from Task 30, which is the simplified year-end/annual-close variant. *(2026-03-21)\_

- **Payment `payment_registered` check** ŌĆö passes when `amountOutstanding = 0`. Must pay the `amountOutstanding` field (VAT-inclusive remaining balance), NOT the `amount` field or the chain-returned amount. PaymentHandler now GETs `/invoice/{id}?fields=id,amount,amountOutstanding` and uses `amountOutstanding` for `paidAmount`. _(2026-03-20, CORRECTED)_
- **Payment has 3 prompt sub-types**: (A) Full chain with order lines ŌĆö create customerŌåÆorderŌåÆinvoiceŌåÆpay; (B) Simple pay ŌĆö customer has pending unpaid invoice, find it and pay `amountOutstanding`; (C) Reversal ŌĆö payment returned by bank, reverse via credit note. Each has different competition checks (2ŌĆō5 checks, 7ŌĆō8 pts max). Handler detects variant by: `action=reverse` ŌåÆ reversal, `hasOrderLines` ŌåÆ full chain, else ŌåÆ simple pay with full-chain fallback. _(2026-03-20)_
- **LLM extracts `orgNumber` not `organizationNumber`** ŌĆö both InvoiceHandler and PaymentHandler must check both key names when looking up customers. Fixed in `ResolveOrCreateCustomer` and `FindCustomerId`. SupplierHandler also affected ŌĆö added fallback from `orgNumber` to `organizationNumber` in POST body. _(2026-03-20, UPDATED)_
- **`vatIncluded: false` does NOT mean 0% VAT** ŌĆö it means the stated amount EXCLUDES VAT. VAT should still be applied at 25%. Use `unitPriceExcludingVatCurrency` with the amount as-is. When `vatIncluded: true`, use `unitPriceIncludingVatCurrency` so Tripletex calculates the ex-VAT amount from the incl-VAT price. Previous code incorrectly switched to 0% VAT type when `vatIncluded: false`, causing wrong invoice totals and failed competition checks. _(2026-03-20, CORRECTED)_
- **Admin role is worth 5 out of 11 points** for employee tasks ŌĆö nearly 50%. Always grant when requested. Verified historical submission logs show `PUT /employee/entitlement/:grantEntitlementsByTemplate?template=ALL_PRIVILEGES` returning 200 for real runs; the raw-prompt admin detection is the safer place to harden this task. _(2026-03-20, UPDATED 2026-03-22)_
- **Employee sandbox validation was under-checking PDF onboarding tasks** ŌĆö `ValidateEmployee` originally only covered base card fields (name/email/DOB), which masked missing department/employment details locally while competition still failed. It now validates department, employment start date, percentage, salary, employment type/form, remuneration type, and occupation-code presence. _(2026-03-21)_
- **Occupation code: STYRK-08 ŌēĀ STYRK-98, use nameNO search** ŌĆö Prompts use 4-digit STYRK-08 codes (e.g. "3323") but Tripletex API stores 7-digit STYRK-98 codes (e.g. "3432101"). These are completely different numbering systems ŌĆö no prefix/substring relationship. `code=3323` returns 0 results because "3323" is NOT a substring of "3432101". Solution: when `code=` search fails, use `GET /employee/employment/occupationCode?nameNO=<name>` to search by Norwegian occupation name. A static `Styrk08ToName` dictionary maps common STYRK-08 codes to Norwegian names (e.g. "3323"ŌåÆ"regnskapsf├Ėrer"). The `nameNO` search found "2511102 AUTORISERT REGNSKAPSF├śRER". Removed wasteful full-scan (was 8 GETs, found nothing). Now: 1 GET (code) + 1 GET (nameNO) = 2 GETs max. Score went from 13/14 (92%) to 14/14 (100%). _(2026-03-21, UPDATED)_
- **Inline employment requires division in sandbox** ŌĆö `POST /employee` with `employments: [{startDate}]` returns 422 "employments.division.id required" in sandbox. Fixed by pre-fetching division (GET /division, free) and including it in the inline employment object. Competition may not require this but it's harmless. _(2026-03-21)_
- **Pre-check email avoids 422 cascade** ŌĆö Sandbox reuses state, so duplicate email triggers 422. Old flow: POSTŌåÆ422ŌåÆPOSTŌåÆ422ŌåÆGETŌåÆreuse (2 errors, 2 wasted writes). New flow: GET email first (free)ŌåÆreuse if found (0 errors, 0 wasted writes). Also works for competition (GET returns empty, proceed to POST). _(2026-03-21)_
- **Product `number` field is checked** ŌĆö validator verifies the product number matches the prompt. Always extract and set it. _(2026-03-20)_
- **Product price key alias** ŌĆö LLM extracts price as `"price"` or `"unitPrice"` key. Handler must alias these to `priceExcludingVatCurrency`. SandboxValidator also needs alias check on the entity dict (uses `"price"` ŌåÆ `"priceExcludingVatCurrency"`). _(2026-03-20)_
- **Product number key alias** ŌĆö LLM extracts number as `"productNumber"` key. Handler aliases it to `"number"` via `SetWithAlias`. SandboxValidator must check `"productNumber"` as alias for `"number"`. _(2026-03-20)_
- **Product vatType sandbox rejection** ŌĆö local sandbox rejects ALL vatType IDs for POST /product with "Ugyldig mva-kode". Competition sandbox accepts vatType=1. Do NOT default vatType unless explicitly extracted. If extracted and fails 422, retry without vatType. _(2026-03-20)_
- **Department validation only checks `department_found`** (2 pts) ŌĆö no field-level checks beyond existence. _(2026-03-20)_
- **Travel expense `has_costs` check** wants `> 0` cost lines ŌĆö at least one `POST /travelExpense/cost` required. _(2026-03-20)_
- **Travel expense has 6 checks worth 8 points total** ŌĆö 6/6 passed with correct employee, title, and all cost lines including per diem. _(2026-03-20)_
- **Travel expense `costItems` key** ŌĆö LLM extracts cost items as `costItems` (camelCase). Handler must check `costs`, `costItems`, `cost_items`, and `costLines` variants. _(2026-03-20)_
- **Travel expense employee resolution** ŌĆö use `extracted.Entities["employee"]` (has firstName/lastName/email), NOT `extracted.Relationships["employee"]` (often contains just the email as a flat string). Search by firstName+lastName first, then by email. ALSO: LLM sometimes nests employee inside `travelExpense` entity instead of as a top-level entity ŌĆö handler must parse nested `travel["employee"]` JsonElement to extract firstName/lastName/email. If still not found, CREATE the employee instead of falling back to "first available" (which links to the wrong person ŌåÆ 0/6). _(2026-03-20, UPDATED)_
- **Travel expense per diem** ŌĆö when `durationDays` and `dailyAllowanceRate` are in travelDetails, generate a per diem cost line (days ├Ś rate). This is a separate cost from explicit expense items. LLM may use `perDiemRate` or `dailyRate` instead of `dailyAllowanceRate` ŌĆö handler checks all three aliases. _(2026-03-20, UPDATED)_
- **Invoice validation checks**: `invoice_found`, `has_customer`, `has_amount > 0`. Amount includes VAT. _(2026-03-20)_
- **Order line `count` defaults to 0 in Tripletex** ŌĆö if you omit `count` on an order line, Tripletex treats it as 0 and the invoice amount becomes 0. Always set `count = 1` as default. _(2026-03-20)_
- **Composite invoice tasks (project + time + invoice)** ŌĆö competition checks 8 points across 4 checks: project/timesheet + invoice. Our local validator only checks invoice (5/5). Must create project, link activity, register timesheet hours, AND create invoice. _(2026-03-20)_
- **Credit note validation**: only checks `credit_note_created` (3 pts). _(2026-03-20)_
- **Voucher account lock detection requires `vatLocked` field** ŌĆö use `fields=id,number,vatLocked,vatType(id,number)` when fetching account. The `vatLocked` boolean is the only reliable way to determine if an account is locked to its vatType. Having a default `vatType` does NOT mean locked (e.g. account 6540 has default vatType 0 but `vatLocked=false` ŌåÆ can override with input VAT). Account 7100 has `vatLocked=true` ŌåÆ must respect its locked vatType. Previous bug: assuming any vatType presence = locked caused supplier invoices on account 6540 to skip VAT lookup entirely, resulting in 0/4 competition score. _(2026-03-20)_
- **Delete entity validation**: only checks `entity_deleted` (3 pts). _(2026-03-20)_
- **All 14 validation tasks achieved 100% correctness** in latest sandbox run. _(2026-03-20)_
- **Payroll STJ serialization fix** ŌĆö anonymous types inside `object` parameters (e.g., payslip body with `employee = new { id = X }`) silently dropped properties during `JsonSerializer.Serialize(body)`. Converting to `Dictionary<string, object>` throughout PayrollHandler fixed `has_employee_link` and `payslip_generated` checks (0/4 ŌåÆ 8/8). Same fix applied to EmployeeHandler entity refs. Always use Dictionary, never anonymous types, for nested objects inside Dictionary/object parameters. _(2026-03-21)_

- **Supplier invoice vatLocked=true+vatType=0 needs manual VAT split** ŌĆö when an expense account (e.g. 7100) has `vatLocked=true` and `vatType={id:0}`, the account rejects any vatType on postings. If the prompt specifies a VAT rate (e.g. 25%), `HandleSupplierInvoice` must create a 3-posting voucher: debit expense(NET, no vatType) + debit input VAT 2710(VAT amount) + credit 2400(-GROSS). Net = gross/(1+rate), VAT = gross-net. Without this, the full gross goes to expense with zero VAT ŌåÆ all checks fail. _(2026-03-22)_
- **Tripletex email rejects non-ASCII chars** ŌĆö `POST /employee` with email containing ├Ė, ├ź, ├” etc. returns 422 "Ugyldig format". `EmployeeHandler.SanitizeEmailPart` now normalizes Nordic chars (├ĖŌåÆo, ├”ŌåÆae, ├źŌåÆa) via `NormalizeToAscii` before filtering to ASCII a-z/0-9. Applied both to synthetic emails and prompt-extracted emails. _(2026-03-22)_
- **Multi-VAT invoice product reference conflict** ŌĆö when a pre-existing product's stored vatType differs from the order line's needed vatType, Tripletex uses the product's vatType, causing wrong amounts. `InvoiceHandler.CreateProductsForLines` now detects multi-VAT invoices (>1 distinct vatType across lines), fetches product vatType, and skips product reference when there's a mismatch. _(2026-03-22)_
- **Reminder fee partialPaymentAmount ignored** ŌĆö `PaymentHandler.HandleReminderFeesAsync` used `ParseDecimalField(payment, "amount")` first, which returned the full `amountOutstanding` (e.g. 31625), before checking `reminderFee.partialPaymentAmount` (e.g. 5000). Fixed: check `partialPaymentAmount` first from both `payment` and `reminderFee` entities. _(2026-03-22)_
- **Ledger correction missing_vat posted to same account** ŌĆö `FindAndGetCounter` resolved counterId to the same account as vatId (both 2710), so both postings netted to zero. Fixed: credit posting now goes to the expense account (c.Account, e.g. 6500), not the counter-account. _(2026-03-22)_

## Competition vs Sandbox Differences

- **Competition runs against a clean/fresh Tripletex instance** ŌĆö no pre-existing data, no duplicate emails or department numbers from prior runs. _(2026-03-20)_
- **Sandbox accumulates data** ŌĆö duplicate emails, department numbers 1ŌĆō7+ already taken. Don't add sandbox-specific workarounds. _(2026-03-20)_
- **Bank account issue appeared only in competition** ŌĆö sandbox had it pre-configured. May need to handle this for invoice tasks. _(2026-03-20)_
- **Competition proxy blocks certain endpoints** ŌĆö `POST /incomingInvoice` returns 403 "no permission" even with KOMPLETT module active. The proxy likely uses an endpoint allowlist.
